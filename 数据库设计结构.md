# 数据库设计结构

# 📊 《Floo\! \(飞路一下\)》数据库设计文档 \(v2\.0 改名版\)

## 数据库总体设计概述 \(Overview\)

- **数据库名：** `floo_core_db`

- **字符集编码：** `utf8mb4`（完美支持外语冷僻字、多国语言及魔法 Emoji）。

- **命名逻辑演进：** \* 原 *知识卡片* 升级为 **Floo 魔法粉末/卡片 \(****`floo_powders`****\)**。

    - 原 *记忆曲线进度* 升级为 **飞路穿梭轨道 \(****`floo_tracks`****\)**。

    - 原 *用户收藏* 升级为 **用户的私人壁炉/收藏夹 \(****`user_fireplaces`****\)**。



## 数据表详细设计清单 \(Data Dictionary\)

### 📄 表1：用户主表 \(`user_main`\)

- **对应功能：** `Step1：用户登录`

- **业务逻辑：** 存储用户账户的核心凭证及基础注册信息。

### 📄 表2：用户学习偏好表 \(`user_learning_preference`\)

- **对应功能：** `Step2：选择模式` ＆ `Step3：用户选择想学啥`

- **业务逻辑：** 记录用户当前激活的难度级别以及所选的学习主题。

### 📄 表3：学习内容库表 \(`learning_contents`\)

- **对应功能：** `Step4：每天AI生成推送` ＆ `Step8：自定义学习内容`

- **业务逻辑：** 集中存储系统AI每日生成的标准内容以及用户自主提交的个性化内容。

### 📄 表4：用户记忆进度追踪表 \(`user_memory_progress`\)

- **对应功能：** `Step5：默写能力` ＆ `Step9：记忆曲线复习`

- **业务逻辑：** 基于艾宾浩斯记忆曲线算法，周期性计算用户对各个内容节点的复习状态及复习时间。

### 📄 表5：每日学习打卡表 \(`user_checkin_records`\)

- **对应功能：** `Step6：学习打卡日历`

- **业务逻辑：** 存储用户按天生成的学习打卡数据。用于前端日历热力图的直接渲染。

### 📄 表6：用户积分账户表 \(`user_point_account`\)

- **对应功能：** `Step7：积分兑换`

- **业务逻辑：** 维系用户积分资产的全局账户平衡。

### 📄 表7：积分流水表 \(`point_log_history`\)

- **对应功能：** `Step7：积分兑换`

- **业务逻辑：** 记录积分的所有流水变化明细。

### 📄 表8：用户默写历史流水表 \(`user_dictation_history`\)

- **对应功能：** `Step5：默写能力`

- **业务逻辑：** 增量记录用户对每一个知识点的每一次默写尝试、耗时以及最终的文本比对结果。通过此表可以分析出用户的高频出错词。

### 📄 表9：每周学习数据统计表 \(`user_weekly_summary`\)

- **对应功能：** `新增功能：每周学习情况 ＆ 周复习`

- **业务逻辑：** 按周（如 2026年第23周，对应字段 `year_week` = `202623`）自动跑批或实时聚合用户的学习状态。用于生成每周复习卡片的判定基准，以及触发周复习时的待复习池。

### 🗂️ 业务集群一：用户与偏好设定（基础设施）

- **核心表：** `user_main`（用户主表） $\rightarrow$ `user_learning_preference`（用户学习偏好表）

- **关联关系：** `1 : 1`

- **数据流向：**

    - 用户登录（Step 1）在 `user_main` 生成记录。

    - 随后选择的难度（Step 2）和主题（Step 3）直接写入 `user_learning_preference`。

    - **给 Claude 的提示：** 每次系统要为用户推送 AI 内容时，必须先读取偏好表的 `difficulty_level` 和 `theme_type` 作为大模型的 Prompt 输入参数。

### 🗂️ 业务集群二：内容生产与个性化（内容源）

- **核心表：** `learning_contents`（学习内容库表）

- **业务逻辑：**

    - **每日 AI 推送（Step 4）：** 后台脚本批量生成数据插入此表，`creator_type = 0`，`user_id = 0`。

    - **用户自定义（Step 8）：** 用户自己输入内容插入此表，`creator_type = 1`，`user_id = 具体用户ID`。

    - **核心桥梁：** 该表的 `id`（即 `content_id`）是后续所有学习、默写、复习行为的**唯一锚点**。

### 🗂️ 业务集群三：学习行为与记忆追踪（产品的核心骨架）

这是最核心的部分，由 4 张表联动，完美闭环了你的 **Step 5（默写）、Step 6（打卡）、Step 9（复习）以及周总结功能**。

1. **日常推题与复习（Step 4 \& Step 9）：**

    - `user_memory_progress` 是中央调度盘。系统每天通过 `next_review_at <= 当前时间` 捞出该用户今天该复习的 `content_id`。

2. **默写行为的增量沉淀（Step 5）：**

    - 用户每次完成默写，数据直接**增量写入** `user_dictation_history`（历史流水表）。

    - 同时，系统计算出的最新正确率，会**覆盖更新**到 `user_memory_progress` 的 `last_accuracy` 中，并根据对错自动计算下一次复习时间（更新 `next_review_at` 和 `review_stage`）。

3. **打卡与周报的异步聚合（Step 6 \+ 周 summary）：**

    - 当天只要完成规定任务，`user_checkin_records` 插入一条 YYYY\-MM\-DD 的打卡记录。

    - **每周情况统计：** 定时任务（或者用户打开周报时）会去定时扫描 `user_dictation_history`（算本周平均正确率）和 `user_checkin_records`（算本周打卡几天），然后将结果封装进 `user_weekly_summary`。

    - **周复习功能：** 当周报触发 `weekly_review_status = 1`（进行中）时，系统直接去 `user_dictation_history` 里捞出本周 `accuracy_rate` 最低的几个 `content_id` 扔给用户去复习。

### 🗂️ 业务集群四：游戏化激励（经济系统）

- **核心表：** `user_point_account`（账户总额表） $\rightarrow$ `point_log_history`（变动流水表）

- **关联关系：** `1 : N`

- **数据流向（Step 7 积分兑换）：**

    - 当**集群三**的行为触发成功（如：`user_checkin_records` 成功插入一条打卡，或者 `user_dictation_history` 产生了一条满分记录）。

    - **触发加分：** `point_log_history` 增加一条正分流水（如 `+10`），同时 `user_point_account` 的 `available_points` 字段加上 10。

    - **触发兑换：** 用户消耗积分，`point_log_history` 增加一条负分流水（如 `-100`），同时 `user_point_account` 扣减 100。

### 🗂️ 业务集群六：书籍精读（白名单授权）

用于导入外部经典读物（例如 tingroom 的《Rich Dad Poor Dad》），只对已授权用户开放。

- **核心表：** `book_series` → `book_chapter` → `book_chapter_segment` + `user_book_access`

- **关联关系：**
    - `book_series 1 : N book_chapter`
    - `book_chapter 1 : N book_chapter_segment`
    - `book_chapter.content_id → learning_contents.content_id`（整章模式）
    - `book_chapter_segment.content_id → learning_contents.content_id`（分段模式，每段 ~450 词）
    - `user_book_access` 是用户-书籍多对多关系表，`is_active` 支持软撤销

- **数据流向：**

    - **导入（一次性）：** admin 调 `POST /api/book/admin/import`，`book_importer.py` 从
      tingroom 抓取目录 + 每章正文 + 页脚编号词汇，落到 `learning_contents`（整章一条、
      每段一条）+ `book_chapter`(_segment)。`book_series.source_url` 是幂等键。

    - **授权：** admin 调 `POST /api/book/admin/grant` 往 `user_book_access` 插一条。

    - **阅读：** 用户端 `/api/book/mine` 走 `user_book_access` 索引查可访问书籍；
      章节页选整章或分段模式，点击后跳 `/pages/detail`，复用已有词典/朗读/默写。

- **为什么不复用 learning_contents 直接建"书籍"关系：**
  书籍需要「顺序 + 授权」两层语义，而 `learning_contents` 只是学习内容的孤立锚点。
  单独 3 张关系表让整章/分段模式和授权闸门都在同一数据结构上完成，
  查询路径清晰、迁移风险小。
